王者荣耀智能客服知识库系统 - 项目答辩模块功能说明

一、项目整体介绍

本项目是一个面向王者荣耀客服场景的智能知识库问答系统，支持用户登录、角色权限控制、知识库管理、文档上传、文本切片、向量检索、Ollama 本地大模型增强回答、对话历史管理、日志分析以及系统参数配置。

系统采用前后端分离式结构，前端页面位于 static 目录，主要包括登录页 login.html 和主页面 index.html；后端核心逻辑集中在 app.py，通过 Flask 提供接口服务。系统默认运行地址为 http://127.0.0.1:8000/，并设置为优先进入登录页面。


二、登录注册与角色权限模块

1. 登录注册功能

系统提供独立的登录页面 static/login.html。用户进入程序后首先访问登录界面，需要输入账号、密码，并选择对应角色进行登录。

登录后系统会保存认证 token，用于后续接口访问校验。用户刷新页面时会自动检查登录状态，避免未登录用户直接访问后台功能。

2. 角色划分

系统内置三种角色：

超级管理员：
拥有全部权限，可以访问对话、知识库管理、向量管理、系统管理、权限管理、日志分析等所有功能。

知识库管理员：
可以访问对话界面、知识库管理、文档管理、向量管理等功能，主要负责知识库内容维护和索引管理。

普通用户：
只能访问对话咨询界面，不能看到左侧后台管理导航，也不能操作知识库、系统配置、日志和权限管理。

3. 权限控制功能

系统后端通过接口权限表对不同 API 进行权限校验。例如：

chat:use：使用对话机器人。
kb:read：查看知识库。
kb:write：管理知识库。
doc:read：查看文档。
doc:write：管理文档。
vector:manage：管理向量。
system:manage：管理系统配置。
auth:manage：管理用户和角色。
log:read：查看日志。

前端也会根据当前用户权限动态隐藏不允许访问的页面和按钮。


三、普通用户对话界面模块

1. Codex 风格对话界面

普通用户登录后进入简洁的对话界面，不显示后台黑色导航栏。界面左侧显示历史对话搜索框、开启新任务按钮和历史对话列表；中间区域为主要聊天窗口；底部为输入框。

2. 对话输入框

输入框采用圆角浮动样式，支持多行输入。发送按钮在输入和生成过程中会有不同状态，生成时按钮会进入旋转或忙碌状态，回答结束后恢复正常。

3. 打字机输出效果

AI 回答不是一次性显示，而是像真实聊天助手一样逐字输出。输出过程中中间对话区域会自动向下滚动，但不会影响左侧历史导航栏。

4. 快捷提问功能

用户对话界面提供快捷提问按钮。初始页面会显示常用问题，例如英雄玩法、出装、账号问题、充值问题等。AI 回答完成后，回答下方还会生成与当前问题相关的继续提问按钮，方便用户连续追问。

5. 回答排版优化

系统会自动清理文档中的 # 符号，避免把 txt 或 markdown 中的标题符号直接显示给用户。

普通回答默认不强制使用 Markdown。只有用户明确要求 Markdown 格式时，才按 Markdown 输出。普通情况下系统会使用自然对话文本，并在前端自动优化显示效果，例如：

首段加粗显示。
“直接结论：”“出装选择：”“注意事项：”等标签自动加粗。
单独一行的小标题会自动放大并加粗。


四、对话历史与上下文记忆模块

1. 用户对话隔离

每个用户都有独立的对话历史，不同账号之间的历史记录互不混合。普通用户只能看到自己的历史对话。

2. 历史对话搜索

普通用户左侧历史区域支持搜索历史记录，可以按照问题标题或内容快速查找以前的对话。

3. 开启新任务

用户可以点击“开启新任务”创建新会话，新的会话不会污染旧会话上下文。

4. 上下文记忆

系统会把最近多轮对话整理成紧凑记忆，传给后端检索和 Ollama 生成逻辑，用于理解“继续说”“具体怎么玩”“那出装呢”等追问。

例如：

第一轮问“韩信怎么玩”。
第二轮问“应该怎么出装”。
系统会理解为“韩信应该怎么出装”。

5. 历史问题查询

系统可以直接回答用户关于历史对话的问题，例如：

“我刚开始问了什么问题？”
“我最后问了什么问题？”
“上一个问题再回答一遍。”

这类问题不会再错误进入知识库检索，而是优先根据对话历史直接回答。

6. 问候识别

系统能识别“你好”“你好嘛”“在吗”等普通问候，不会把这类短句强行关联到上一个英雄或知识库主题。

7. 复制和更新对话

用户消息旁边提供复制和编辑按钮。点击复制后会出现弹窗提示，并支持基于复制的问题发起新对话。

点击编辑可以修改某条用户提问，并让 AI 根据新问题重新生成回答。更新后的对话会追加到当前会话底部，不会影响其他历史对话记录。


五、知识库管理模块

1. 创建知识库

知识库管理员和超级管理员可以创建知识库。创建时可以填写：

知识库名称。
所属部门。
分类。
负责人。
Embedding 模型。
知识库描述。

2. 知识库分类

系统默认提供多个分类，例如：

英雄攻略。
活动规则。
账号问题。
充值答疑。
故障排查。

3. 知识库列表

系统提供知识库列表，可以查看所有知识库，并支持按照名称、描述、负责人、部门进行筛选。也可以按照创建时间、名称、部门排序。

4. 编辑知识库

管理员可以编辑知识库名称、部门、负责人、分类、Embedding 模型和描述。

5. 删除知识库

系统支持知识库删除功能。删除知识库时会同步处理相关文档和索引，避免前端知识库列表中出现无法删除的问题。

6. 克隆知识库

系统支持克隆知识库，可以复制已有知识库结构和内容，便于快速创建相似场景知识库。


六、文档上传与解析模块

1. 文档上传

知识库管理员可以向指定知识库上传文档。系统支持多文件上传。

支持格式包括：

txt。
md。
docx。
pdf。

2. 自动解析

上传后系统会自动解析文档内容。如果解析失败，会记录失败状态和原因。

3. 自动清洗

系统会对上传文本进行清洗，去除多余空白和不适合显示的符号，保证后续切片和展示效果更稳定。

4. 自动分块

系统会按照配置的 chunkSize 和 chunkOverlap 对文档进行切片。当前默认配置为：

Chunk Size：400。
Overlap：50。

这样可以让每个文本块保留足够上下文，减少回答不完整的问题。

5. 文档列表

系统提供文档列表，可以查看知识库中的文档名称、状态、上传时间等信息。

6. 文档预览与编辑

管理员可以打开文档预览，查看原文和分块结果。也可以直接修改文档内容，保存后系统会重新向量化。

7. 批量删除文档

系统支持批量删除文档及对应向量索引，方便维护知识库内容。


七、向量管理模块

1. 自动向量化

文档上传后会自动进行文本切片和向量索引建立。系统使用本地关键词向量和可配置的 Embedding 模型完成检索索引。

2. 手动重新生成向量

当修改了 chunkSize、chunkOverlap 或文档内容后，可以在向量管理页面手动重新生成向量索引。

3. 向量归一化

系统支持对向量数据进行归一化处理，提高不同文本块之间相似度计算的稳定性。

4. 向量去重

系统支持按照相似度阈值删除重复文本块索引，减少冗余数据，提高检索效率。

5. 向量预览

管理员可以输入关键词查看向量预览结果，包括：

文本块编号。
所属知识库。
所属文档。
模型名称。
向量维度。
范数。
相似度。
关键词项预览。


八、知识库测试工具模块

1. 测试目的

知识库测试工具用于排查“上传了知识库但对话问不到”的问题。

它只执行检索，不调用 AI 生成回答，也不会写入对话历史。

2. 测试内容

管理员可以在向量管理页面输入一个测试问题，选择知识库和 Top K，系统会返回：

有效问题。
实际检索词。
回答模式。
可靠阈值。
最高分。
是否命中可靠知识库。
命中的文档和文本片段。

3. 排查价值

如果测试工具没有召回文本块，说明问题可能出在：

知识库选择错误。
文档没有成功上传。
文档没有完成向量化。
用户问题关键词和文档内容差异太大。

如果测试工具能召回，但对话回答不准，说明问题可能出在：

LLM 生成阶段偏离。
阈值设置不合理。
上下文改写不准确。


九、命中调试面板模块

1. 对话调试信息

AI 回答下方会显示可展开的“命中调试”区域。

2. 调试内容

命中调试面板包含：

回答来源：知识库、自由生成、兜底话术或直接规则。
当前回答模式。
用户原始问题。
系统改写后的有效问题。
实际检索词。
可靠阈值。
最高命中分数。
检索状态。
命中的知识库、文档和文本片段。

3. 功能意义

该功能可以让管理员明确知道 AI 为什么这样回答，便于快速定位问题是出在知识库内容、检索召回、阈值配置还是大模型生成。


十、三种回答模式模块

系统支持三种回答模式，可以在系统管理中配置。

1. 严格知识库模式

只有当知识库检索达到可靠阈值时，系统才回答。

如果没有可靠命中，系统会返回兜底话术，例如“未在知识库中找到相关答案，请换个问题试试。”

该模式适合客服、规章制度、业务规则等要求准确可靠的场景。

2. 知识库增强模式

当知识库可靠命中后，系统会把检索结果交给本地 Ollama 模型进行自然语言增强，让回答更加流畅。

如果没有可靠命中，则不会乱编知识库来源。

这是系统默认推荐模式，兼顾准确性和自然表达。

3. 自由问答模式

当知识库没有可靠命中时，系统允许 Ollama 根据通用知识回答。

但自由回答不会标注为知识库来源，避免出现“知识库没有内容却显示来源知识库”的问题。

该模式适合演示开放问答能力，但准确性不如严格知识库模式。


十一、Ollama 本地大模型增强模块

1. 本地模型调用

系统支持调用本地部署的 Ollama 服务，默认地址为：

http://127.0.0.1:11434

2. 本地模型检测

系统可以检测本地 Ollama 已部署的模型，并在下拉框中显示出来，管理员可以选择不同模型进行切换。

3. LLM 增强回答

当开启 Ollama 增强后，系统会把检索到的知识库片段和上下文记忆传给模型，让模型基于知识库生成更自然的回答。

4. 防止乱编

在知识库增强模式下，提示词要求模型严格基于检索资料回答。如果资料不足，需要说明资料不足，而不是编造。

5. 回答格式控制

系统提示词要求：

用户没有明确要求 Markdown 时，默认使用普通自然对话文本。
用户明确要求 Markdown、表格或列表时，才按用户要求输出。
回答不能以省略号结尾。
回答需要完整收尾。

6. 在线 API 模型切换

除了本地 Ollama，系统也支持配置在线 API 作为增强模型来源。超级管理员可以在系统管理中选择模型提供方，并填写厂商名称、API 地址、模型名称和 API Key。

在线 API 按 OpenAI 兼容的 chat/completions 接口调用，适合接入 DeepSeek、OpenAI、Kimi、通义千问等兼容服务。

切换后，对话增强逻辑会根据当前配置自动选择本地 Ollama 或在线 API，不需要普通用户修改使用方式。


十二、检索与回答生成模块

1. 查询改写

系统会根据上下文对用户问题进行改写。例如：

用户问“韩信怎么玩”，下一轮问“应该怎么出装”，系统会改写成“韩信 应该怎么出装”。

2. 英雄识别

系统支持识别多个王者荣耀英雄名称，并处理部分别名，例如：

“凯”会规范为“铠”。
“鲁班”会识别为“鲁班七号”。
“后弈”会识别为“后羿”。

3. 检索排序

系统会对知识库文本块进行打分，并结合问题类型调整权重，例如：

出装问题优先匹配出装、装备、铭文内容。
克制问题优先匹配克制关系内容。
连招问题优先匹配技能连招内容。
英雄问题优先保留相关英雄文本块。

4. 答案拼接

系统会从高分文本块中拼接答案，并限制最大回答长度，避免回答过长或被截断。

5. 兜底回答

当没有可靠知识库命中时，系统会返回后台配置的兜底话术。


十三、违禁词与拒答规则模块

1. 敏感词过滤

系统支持配置敏感词和违禁词。用户问题或回答中命中敏感词时，会自动用星号替换。

2. TXT 导入违禁词

管理员可以上传 txt 文件导入违禁词。建议每行一个词，导入后点击保存配置即可生效。

3. 拒答关键词

系统支持配置拒答关键词，例如违法、攻击、色情、外挂、盗号等。

4. 拒答话术

当用户问题命中拒答关键词时，系统不会进入知识库检索，而是直接返回拒答话术。

5. 固定对话规则

系统支持配置固定规则，例如：

帮助=你可以询问账号问题、充值到账、活动奖励、英雄玩法等。
版本=当前系统为王者荣耀客服知识库演示版。

当用户问题命中规则关键词时，系统直接返回固定回答。


十四、系统配置模块

超级管理员可以在系统管理中配置多项参数：

默认 Chunk Size。
默认 Overlap。
上传文件大小上限。
接口超时时间。
默认 Top K。
默认相似度阈值。
最低可靠分数。
答案最大字数。
上下文轮数。
上下文最大字数。
检索超时秒数。
回答生成超时秒数。
回答模式。
是否开启 Ollama 增强。
Ollama 地址。
Ollama 模型。
模型温度。
在线 API 厂商名称。
在线 API 地址。
在线 API 模型名称。
在线 API Key。
缓存策略。
兜底话术。
敏感词。
拒答关键词。
固定对话规则。

这些参数保存后会影响后续上传、检索和回答生成流程。

系统管理页面采用左右分栏布局。左侧为模型配置，右侧为系统参数配置。两个分栏都支持独立上下滚动，互不影响。

后台主导航栏固定在屏幕左侧，左下角账号信息始终显示在屏幕左下角，方便管理员在配置大量参数时随时确认当前登录账号并执行退出操作。


十五、模型配置模块

1. Embedding 模型管理

系统支持添加 Embedding 模型，包括：

模型名称。
模型路径。
向量维度。
模型描述。
是否设为默认模型。

2. 默认模型设置

管理员可以将某个 Embedding 模型设为默认模型，后续创建知识库或向量化时可以使用该模型。

3. 本地模型发现

系统会扫描本地 models 目录和 data/models 目录，发现可用模型并加入模型列表。


十六、用户个人设置模块

1. 用户头像

用户可以在左下角点击头像菜单，进入设置页面上传头像。

支持图片类型包括：

png。
jpg。
jpeg。
webp。
gif。

2. 昵称修改

用户可以修改显示名称，名称会显示在头像右侧。

3. 用户菜单

点击左下角头像后，会向上弹出白色圆角悬浮菜单，带轻微阴影。菜单包含：

设置。
收藏夹。
切换账号。
退出登录。


十七、日志分析模块

1. 对话日志记录

每次用户提问和 AI 回答都会写入日志，包括：

会话 ID。
用户 ID。
问题。
答案。
知识库 ID。
来源。
最高相似度分数。
耗时。
创建时间。

2. 按账号收纳

日志分析页面会按照不同账号分组显示，每个账号下的对话日志可以收起和展开。

3. 日志搜索

日志页面支持搜索账号、问题、答案、知识库名称和会话 ID，方便管理员排查用户问题。

4. 权限控制

只有拥有 log:read 权限的用户才能查看日志分析页面。普通用户不能访问日志。

1. 登录页面

登录页独立存在，地址为：

static/login.html

页面提供角色选择登录，包括超级管理员、知识库管理员和普通用户。

2. 主页面

主页面为：

static/index.html

根据用户权限显示不同功能模块。

3. 样式文件

主要样式文件为：

static/styles.css

5. 系统管理滚动布局

系统管理页面针对参数较多的问题做了独立滚动设计：

页面整体固定为浏览器高度。
左侧黑色后台导航栏不会随右侧内容滚动。
左侧导航内部可以单独滚动。
左下角账号信息固定显示。
系统管理的左右两个配置面板可以分别滚动，互不影响。

这样管理员在修改模型、知识库参数、敏感词、在线 API 等配置时，不会因为右侧表单过长导致左侧账号信息消失，也不会出现左右区域互相带动滚动的问题。

二十一、项目特色与创新点

1. 支持三角色权限体系

系统区分超级管理员、知识库管理员和普通用户，普通用户只能聊天，后台功能不会暴露。

2. 支持知识库增强回答

系统不是简单聊天机器人，而是通过知识库检索加本地大模型增强回答，提高回答准确性和自然度。

3. 支持本地 Ollama 部署

系统可以使用本地大模型，不依赖外部云接口，适合离线演示和本地部署。

4. 支持知识库测试和命中调试

管理员可以清楚看到问题是否命中知识库、命中了哪些文档、相似度是多少，方便排查问题。

5. 支持上下文记忆

系统能理解多轮追问，不会每次都把问题当成孤立问题处理。

6. 支持普通文本和 Markdown 输出控制

用户没有要求 Markdown 时，系统默认使用普通对话文本，前端负责优化显示层级。

7. 支持敏感词和拒答规则

系统可以过滤敏感词，并对违规问题返回拒答话术。

8. 支持日志按账号分组

管理员可以按账号查看和搜索对话日志，方便运营和问题追踪。

9. 支持本地与在线模型灵活切换

系统既可以使用本地 Ollama，也可以配置在线 API，通过厂商名称、API 地址、模型名称和 API Key 切换不同模型，适合本地演示和在线增强两种部署方式。

10. 支持后台管理独立滚动布局

系统管理页面的左右配置面板互不影响，左侧账号信息固定可见，提升了后台大量参数配置时的可用性。


二十二、答辩时可以重点说明的问题

1. 为什么要做知识库测试工具？

因为在实际使用中，经常会出现“文档已经上传，但用户问题没有命中”的情况。测试工具可以单独验证检索阶段，判断问题出在文档、向量化、检索词还是大模型生成。

2. 为什么要做三种回答模式？

不同场景对准确性的要求不同。客服和规则类问题更适合严格知识库模式；演示和普通问答可以使用自由模式；平时推荐知识库增强模式，既能保证依据，又能提高回答自然度。

3. 为什么普通用户看不到后台？

这是权限控制设计。普通用户只需要咨询，不应该接触知识库维护、系统配置和日志数据，避免误操作和数据泄露。

4. 为什么使用本地 Ollama？

本地 Ollama 可以离线部署，保护数据隐私，同时便于在实训环境中演示大模型增强能力。

5. 如何解决上下文问题？

系统会保存最近多轮对话，并对追问进行问题改写。例如用户先问“韩信怎么玩”，再问“怎么出装”，系统会理解为“韩信怎么出装”。

6. 如何避免 AI 乱标来源？

系统把知识库回答和自由回答分开。只有可靠命中知识库时才显示知识库来源；自由模式生成的内容不会伪装成知识库来源。

7. 为什么增加在线 API 切换？

因为不同部署环境可用模型不同。本地部署适合离线演示和隐私保护，在线 API 适合调用更强模型提升回答质量。系统把两种方式统一到同一套配置中，管理员可以按场景切换。

8. 为什么系统管理要做左右独立滚动？

系统管理页参数很多，如果整页一起滚动，左侧导航和账号信息容易离开视野。独立滚动后，管理员可以一边查看当前账号和导航，一边单独滚动右侧配置表单，操作更稳定。

二十四、总结

本项目实现了一个完整的智能客服知识库系统，从用户登录、角色权限、知识库上传、文本向量化、智能检索、LLM 增强回答、上下文记忆、对话日志到系统配置均已形成闭环。

系统不仅能完成普通用户的咨询问答，还提供了知识库管理员和超级管理员所需的维护、调试和分析功能。整体功能适合用于实训项目答辩，能够体现前后端交互、数据库设计、权限控制、知识库检索和大模型应用等综合能力。
